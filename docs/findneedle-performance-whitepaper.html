3. Opening a log

Opening a log reads the file, parses each entry into a record, and stores it. The viewer becomes browsable as soon as the data is stored; the full-text index (§4) then builds in the background by default, so browsing is not blocked on it.

App launch → main window
~3.0 s
Under ~10k rows (in-memory)
<1 s
200,000 rows → viewer
~2.5 s
5,000,000-event .etl → viewer
~36 s
5,000,000-line text log → viewer
~43 s

Large logs (millions of rows) load by inserting across several CPU cores in parallel while one thread reads and parses, which is the default. The file is streamed (never fully buffered in memory) and fields are parsed lazily. Parallel insert helps in the multi-million-row range; for very large logs (tens of millions of rows) the picture changes — see §8 on scaling.

4. Full-text index build

Substring search over large logs is served by a full-text (trigram) index. Building it is the most expensive phase on a large log. By default it builds in the background right after the viewer opens, so the user can browse immediately; substring search becomes instant once it completes.

500,000 rows
~5 s
5,000,000 rows
~15 s
50,000,000 rows
~187 s

The build is super-linear in indexed text (≈12× for 10× more rows), so it dominates the "ready to search" time on the largest logs — see the per-size matrix in §8.

Three index timings are available as a setting: background (build right after the viewer opens — the default), lazy (build on the first search, or never if the user never searches), and eager (build during load, so the first search is instant but the open takes longer — a 5M-row log opens in about 64 s in this mode). The index is reused on reopen (§7), so the build cost is paid once.

5. Search latency

Once the index exists, substring search returns in well under a second even on 5,000,000 rows. If a search is run before the index is ready, the app does a bounded table scan instead of freezing.

Indexed search, 5M rows
~0.7 s
Warm reopen (cached), search
~0.1 s
Un-indexed first search, 5M rows (scan)
~16 s

Indexed-search time depends on how many rows the term matches, not just the log size: a selective term (rare in the log) is sub-second at any size, while a term that matches a large share of rows scales with the match count (on a log of near-identical lines a broad term can take seconds — see the §8 matrix).

6. UI responsiveness & the result viewer

The viewer reads rows from storage a page at a time and never loads the whole result set into memory. Searching and paging run off the UI thread, so the window stays responsive at any log size.

Jumping to the last page of a 5M-row set takes about the same time regardless of how far it is from the first page. The application monitors for any interaction over 2.5 s; across the automated UI suite none occurred.

8. How performance scales with log size

Most of what a user does is designed not to grow with the size of the log — opening a page, searching once indexed, scrolling, and reopening are all near-constant whether the log has thousands or tens of millions of rows. Only the one-time work of ingesting and indexing grows with size.

First-open ingest (read + parse + store)O(n)Linear in rows — the unavoidable one-time cost

Memory footprintO(1)Rows are paged from disk; RAM stays flat at any scale

Serial ingest is cleanly linear — ≈9–10 µs per row at every size. Parallel fan-out is faster across this whole range, but its advantage steadily shrinks (1.28× → 1.09×) because the shard-merge step grows faster than the parallel insert saves.

On real captures the advantage erodes sooner. The series above is uniform synthetic data. A real 36-million-event capture (3.7 GB, ≈90% kernel events, varied payloads) has much heavier rows, so its merge costs far more — there, parallel ingest was actually slower than serial (554 s vs 439 s, 0.79×). So fan-out helps most in the low tens of millions of rows; for the very largest real logs the single-writer serial load is the safer choice.

9. Scoped loading for huge files

Because ingest is O(n) in the rows actually stored — and can't be meaningfully sped up at the top of the curve (the merge even backfires) — the real win for a multi-GB capture is to not load what you don't need. Opening a large log offers a triage step: inspect the file for its provider mix first (a bounded scan — seconds, not minutes), then load only the providers (and/or time window) you pick. The selection becomes a RuleDSL scope rule whose provider/time predicates are evaluated at decode time — dropping the rest before the expensive wrap + insert, so you pay for only the events you keep. This works across formats: ETL and Event Log (.evtx) scope by provider and time window; plain text scopes by time window (lines with no parseable timestamp are always kept). (See docs/scope-rule-design.md.)

Measured on the real 36-million-event, 3.7 GB capture — scoping to the 24 non-kernel providers (dropping the ~90% kernel scheduling firehose), through the actual rule → decode-filter path:

Full load — all 36M rows
408 s
Scoped load — 2.8M rows kept
50 s

8.2× faster by loading 8% of the file. Scoping lets the user choose where on the §8 cost curve they land — e.g. "just the application + .NET events", a specific time window, or a diagram-only pass — instead of waiting on the whole capture. The file is still scanned once to filter, so the decode pass is the floor; everything above it is what you chose to keep.

Speedup tracks the share kept, plus a per-format floor that scope can't skip. Plain text scales almost with the rows stored (3.1× for 22%), because the saving is the avoided storage + index work. Event Log gains less per row dropped (1.7× for dropping 44%) because every record is still read off disk by the Event Log reader before the scope sees it — the win is skipping the costly message render (FormatDescription) and the insert for dropped events. Event Log is also expensive per event in absolute terms (~1.2 ms each), so scoping a noisy channel matters even on a small file. * the 5 M ETL row is the §8 baseline for reference; the scoped figure depends on which providers/window you keep.

10. How these numbers were produced

Every figure is regenerated by the automated test suites and the app's phase-timing log (%LocalAppData%\FindNeedle\perf-log.txt). Key entry points:

  • Search_LazyUnindexed_StaysResponsive / Search_EagerIndex_IsFast — end-to-end load, search latency, and UI round-trip on 5M rows (real app, via FlaUI).
  • FiveMillionRows_AllPageSizes_PaginateToEnd_AndScroll — paging and scroll across page sizes.
  • StorageBackendScale / Cache_Fresh_Then_Hit_Then_Disabled — tier selection and warm reopen.
  • SqliteInsertBenchmark, SearchLatencyBenchmark, EtlParallelIngestIntegrationTests — component timings.
  • TriageScopeTests (ETL), EvtxScopePerfTests (Event Log), PlainTextScopePerfTests (text) — full-vs-scoped load per format (§9).
  • In-app About → Export performance report writes a machine-anchored snapshot (spec, the last open's phase timings, any slow interactions, and cache footprint).

11. Notes

  • Measured on one reference machine (a laptop CPU). Absolute times vary with CPU and SSD; sustained multi-second builds show some thermal variance.
  • Load and search times scale with the number of rows and the length of each entry; logs with very long lines or many columns take longer to index.
  • Text-log and ETL load times differ because they are parsed differently; both are shown in §3.
