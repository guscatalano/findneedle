1. Executive summary

FindNeedle stays interactive on multi-million-row logs by keeping the heavy work off the UI thread and never holding the full result set in memory. The dominant cost on large logs is the full-text index build; this build is now parallelised so a 5-million-row log is searchable in seconds rather than minutes.

  • Ingestion sustains ~115k rows/s; it is write-bound (SQLite insert), not parse-bound.
  • Index build — the single biggest cost — was cut ~48% by trimming redundant indexed text, then parallelised across cores for a further ~5× on large logs.
  • Search is sub-second once indexed; the un-indexed first search degrades gracefully to a bounded scan while a background index builds.
  • The viewer never blocks: results are paged from storage on demand, so the UI thread stays responsive (~12 ms round-trips) regardless of result-set size.
  • Footprint is bounded: storage tier is chosen by size, the on-disk cache self-limits to 10 GB, and temp extraction dirs are cleaned on exit.

3. Methodology

Scenarios are driven by the project's automated test suites and a structured timing log the app emits for every phase (%LocalAppData%\FindNeedle\perf-log.txt):

  • End-to-end (FlaUI UI tests) drive the real app — load a file, search, paginate, scroll — and read timings from the live UI. e.g. SearchResponsivenessUITests, NativeGridLargeScaleUITests (the 5M-row paging/scroll suite).
  • Component benchmarks (MSTest) isolate one mechanism for a fast feedback loop: SqliteInsertBenchmark (insert throughput), FtsShardSpike (single vs parallel index build + query).
  • Synthetic logs use realistic line shapes (~120–210 chars: timestamp, level, message, ids, paths) so string lengths and index sizes resemble production logs.

Caveat. Synthetic data has uniform structure; real logs vary. Index-build ratios were validated on both synthetic and realistic-message data and held within a few points.

4. Data ingestion

Ingestion = read the file → parse each line into a record → insert into SQLite. Measured on 5,000,000 rows. The insert dominates; parsing is comparatively cheap and the file read is sequential.

≈115k rows/s end-to-end (≈160k rows/s for the insert alone). The parser streams the file (no full-file buffering), parses fields lazily, and the insert reuses one prepared statement — so the remaining cost is SQLite's single-writer insert, which is a hard floor for one database.

5. Full-text index build biggest lever

Substring search over large logs is served by a SQLite FTS5 trigram index. Building it is the single most expensive phase, and its cost is super-linear in indexed text. Two rounds of work took a 5M-row build from ~260 s to ~16 s:

Original (all columns, 1 index)
~260 s
+ stop double-indexing text
~160 s
+ drop constant path/timestamp cols
~110 s*
+ parallel shards (8 cores)
~16 s

Column trimming (−48% on a realistic log)

The index covered six columns; several were redundant or low-value. On a 500k-row realistic log the build dropped 39.5 s → 20.5 s by (a) not trigram-indexing the message twice (SearchableData duplicated Message for text logs), and (b) skipping the Source/ResultSource file paths and LogTime when they are a single constant value — a repeated path produces huge, useless trigram lists. Multi-file/folder loads still index paths so path search keeps working; timestamps remain covered by the dedicated time-range filters.

Parallel sharding (~5×)

A single index is built by one writer thread. The build is instead split into N shards (one contentless FTS index per CPU, capped at 8), built concurrently, and queried by attaching the shards and unioning the per-shard matches. Rows stay in one table, so paging/sort/count are unchanged.

Net effect: "ready for fast full-text search" on a 5-million-row log went from minutes to ~16 seconds. The index also warm-reuses across reopens (no rebuild) and is bounded by the same 10 GB cache cap.

6. Search latency

Once the index exists, substring search is effectively instant. Before it exists, the app falls back to a bounded table scan rather than freezing — and builds the index in the background so subsequent searches are fast.

Un-indexed first search (5M LIKE scan)
~15 s
Indexed search (selective term)
<0.1 s
Cross-shard match (worst case, all rows)
~0.2 s

Indexing modes let the user trade open-time for search-time: lazy (skip the index until the first search), background (build right after open, off the critical path — the default), or eager (build during load for an instant first search).

7. UI responsiveness & the result viewer

The viewer reads rows through a paged source — it never materialises the whole result set. Search and paging run off the UI thread, so the window stays interactive no matter how large the log.

Interaction (5,000,000 rows)Time
UI round-trip sampled during a running search~12 ms
Render a page (first page)~0.17 s
Jump to last page (keyset, not deep OFFSET)O(page size)
Pixel-scroll within a pageno UI block

Deep pagination uses a keyset (reverse-sort + limit) instead of a large OFFSET scan, so "jump to the last page" of a 5M-row set is as fast as the first — it does not degrade with offset. A built-in UX monitor records any interaction exceeding 2.5 s; across a full automated suite none did.

8. Storage tier & footprint

The backend is chosen automatically by estimated row count, balancing speed against memory:

Estimated rowsBackendRationale
< 10,000In-memoryFastest; trivial memory
10,000 – 50,000HybridIn-memory working set, settles to disk
> 50,000SQLitePages from disk; constant memory at any scale

If the estimate proves wrong mid-load, the engine falls back to a lower-memory tier rather than exhausting RAM. On disk, completed scans are cached (keyed by source path + size + mtime) so reopening a log skips the scan entirely; the cache self-limits to 10 GB (LRU eviction, enforced after every write), and temp extraction directories (from .zip/.cab) are removed on exit.

A warm reopen validates the cached database (and re-attaches any index shards) and goes straight to the viewer — the scan, parse, and index build are all skipped.

12. Limitations & honest notes

  • Single reference machine (a laptop). Absolute times vary with CPU/SSD; multi-minute builds show thermal variance. Treat ratios as the durable result.
  • Sharded index builds parallelise only line-oriented and already-ingested data; ETL decode is sequential by nature (stateful, single-pass) and is not chunked.
  • SQLite single-writer insert is a floor for ingestion on one database; it was not parallelised (parallelising it would require multi-database row sharding, a larger change).
