3. Opening a log

Opening a log reads the file, parses each entry into a record, and stores it. The viewer becomes browsable as soon as the data is stored; the full-text index (§4) then builds in the background by default, so browsing is not blocked on it.

5,000,000-line text log → viewer

Large logs (millions of rows) load by inserting across several CPU cores in parallel while one thread reads and parses, which is the default. The file is streamed (never fully buffered in memory) and fields are parsed lazily.

4. Full-text index build

Substring search over large logs is served by a full-text (trigram) index. Building it is the most expensive phase on a large log. By default it builds in the background right after the viewer opens, so the user can browse immediately; substring search becomes instant once it completes.

200,000 rows
<1 s
5,000,000 rows
~21 s

Three index timings are available as a setting: background (build right after the viewer opens — the default), lazy (build on the first search, or never if the user never searches), and eager (build during load, so the first search is instant but the open takes longer — a 5M-row log opens in about 64 s in this mode). The index is reused on reopen (§7), so the build cost is paid once.

5. Search latency

Once the index exists, substring search returns in well under a second even on 5,000,000 rows. If a search is run before the index is ready, the app does a bounded table scan instead of freezing.

6. UI responsiveness & the result viewer

The viewer reads rows from storage a page at a time and never loads the whole result set into memory. Searching and paging run off the UI thread, so the window stays responsive at any log size.

Interaction (5,000,000 rows)Expected time

Jumping to the last page of a 5M-row set takes about the same time regardless of how far it is from the first page. The application monitors for any interaction over 2.5 s; across the automated UI suite none occurred.

7. Storage tier & caching

The storage backend is chosen automatically by the estimated row count:

Estimated rowsBackendBehaviour
< 10,000In-memoryFastest; trivial memory use
10,000 – 50,000HybridIn-memory working set, settles to disk
> 50,000SQLitePages from disk; memory stays flat at any scale

A completed open is cached on disk, keyed by the source file. Reopening the same log skips the read, parse, and index build entirely:

The on-disk cache limits itself to 10 GB (least-recently-used eviction), and temporary directories extracted from .zip/.cab archives are removed when the app exits.

8. How these numbers were produced

Every figure is regenerated by the automated test suites and the app's phase-timing log (%LocalAppData%\FindNeedle\perf-log.txt). Key entry points:

  • Search_LazyUnindexed_StaysResponsive / Search_EagerIndex_IsFast — end-to-end load, search latency, and UI round-trip on 5M rows (real app, via FlaUI).
  • FiveMillionRows_AllPageSizes_PaginateToEnd_AndScroll — paging and scroll across page sizes.
  • StorageBackendScale / Cache_Fresh_Then_Hit_Then_Disabled — tier selection and warm reopen.
  • SqliteInsertBenchmark, SearchLatencyBenchmark, EtlParallelIngestIntegrationTests — component timings.
  • In-app About → Export performance report writes a machine-anchored snapshot (spec, the last open's phase timings, any slow interactions, and cache footprint).

9. Notes

  • Measured on one reference machine (a laptop CPU). Absolute times vary with CPU and SSD; sustained multi-second builds show some thermal variance.
  • Load and search times scale with the number of rows and the length of each entry; logs with very long lines or many columns take longer to index.
  • Text-log and ETL load times differ because they are parsed differently; both are shown in §3.
